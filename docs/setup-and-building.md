# Setup and building

## Prerequisites

- A JDK the toolchain can select from — the build requests a per-Minecraft-version JDK via Gradle
  toolchains (see [`buildSrc`](../buildSrc/src/main/java/dev/x341/mrw/BuildTools.java)), so a
  modern JDK (17+) on your machine plus internet access for toolchain provisioning is normally
  enough. No system-wide Gradle install is required or used — always go through the wrapper.
- Nothing else needs installing by hand; Fabric Loom and ForgeGradle pull their own toolchains.

## One repo, two loaders, six Minecraft versions

This is a **single-version-per-build** setup: `minecraftVersion` in the root
[`gradle.properties`](../gradle.properties) picks which Minecraft version you're building against
(currently `1.20.1`). To build a different version, change that property (or pass
`-PminecraftVersion=1.20.4` — quote it in PowerShell, an unquoted value gets truncated to `"1"`)
and rebuild; [`buildSrc/BuildTools.java`](../buildSrc/src/main/java/dev/x341/mrw/BuildTools.java)
is the single table mapping each supported version to its Yarn/Fabric Loader/Fabric API/Forge/Java
release/pack-format coordinates. Supported versions: 1.17.1, 1.18.2, 1.19.2, 1.19.4, 1.20.1, 1.20.4.

Both loaders are separate Gradle subprojects (`fabric/`, `forge/`) built from the *same* Java
source — see [architecture.md](architecture.md#the-fabric--forge-split) for how that works. CI
(`.github/workflows/release-and-build.yml`, `nightly.yml`) builds the full version matrix; locally
you normally only need the one version you're testing against.

## Common commands

Run these from the repo root, using the wrapper (`./gradlew` on Bash, `gradlew.bat` or
`./gradlew` under PowerShell):

```
./gradlew :fabric:compileJava      # fastest correctness check — see testing.md
./gradlew :forge:compileJava       # same, for the Forge copy (regenerated by setupFiles first)
./gradlew build                    # full build, both loaders, produces jars
```

Add `--offline` if you have everything already cached and want to skip network/version checks
(useful when iterating quickly, but won't pick up dependency changes).

Finished jars land in `build/release/` at the repo root (each subproject's `moveBuild` task moves
them there after `build`), named `mtr-rail-works-<loader>-<version>+<minecraftVersion>.jar`.

## Gotchas

- **Forge's `dev.x341.mrw.mod.*` is generated, not hand-written.** The `forge:setupFiles` Gradle
  task deletes and re-copies that entire package (plus `assets`/`data` resources) from the Fabric
  module before compiling. Those copies are gitignored — never edit files under
  `forge/src/main/java/dev/x341/mrw/mod/`, edit the Fabric originals and rebuild. Loader-specific
  bootstrap files (`MRWFabric(Client).java`, `MRWForge.java`, and anything outside the `mod`
  package, e.g. `forge/.../RailWorkerClientHandler.java` or `fabric/.../mixin/MouseScrollMixin.java`)
  are untouched by this and are committed normally.
- **Resource files must be UTF-8, no BOM.** Some editors default to UTF-16LE for JSON, which
  Gson/fabric-loader can't parse at runtime. Check the encoding of any new resource file.
- **Toolchain versions are pinned deliberately** (Gradle wrapper 8.14.x, ForgeGradle 6.0.36, Loom
  1.10-SNAPSHOT) — don't bump these casually. Newer ForgeGradle releases have broken MCP artifact
  generation and Gradle-configuration compatibility in the past; see
  [architecture.md](architecture.md) for why the exact combination matters.
- **MTR itself is a binary dependency, not source.** It's pulled from Modrinth's Maven
  (`maven.modrinth:minecraft-transit-railway:FABRIC-${mtrVersion}+${minecraftVersion}`, `FORGE-...`
  via `fg.deobf` on the Forge side). There is no sources jar; see
  [architecture.md](architecture.md#working-against-a-closed-source-dependency) for how to look up
  its API when you need to.
